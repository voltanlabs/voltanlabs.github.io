# DataByteSprites: Data Discovery Scanner Map

Project tag: `DBD-SCANNER-OS`

Purpose: keep the active Scanner OS app separated from old legacy UI files and prevent wrapper-on-wrapper battle work.

## Phase 1 decision — source of truth

As of this pass, the active live Scanner OS runtime is:

```text
databyte-discovery.html
  ├─ assets/js/databyte-standalone-app.js
  └─ assets/js/dd-scan-bg.js
```

`databyte-discovery.html` is the live boot page. It loads `databyte-standalone-app.js` and `dd-scan-bg.js` only.

`databyte-standalone-app.js` is the canonical app owner for:

- scanner screen
- signal screen
- capture screen
- party screen
- dex screen
- items screen
- admin screen
- encounter state
- capture probability/currentChance
- stability
- collection storage
- seen/dex storage

`dd-scan-bg.js` is the visual/module loader. It may load polish modules, but it is not the owner of encounter flow.

## Critical architecture rule

Do not add more battle overlays as the primary fix.

The battle route must be repaired in the canonical active path:

```text
assets/js/databyte-standalone-app.js
```

A new battle module is allowed only if `databyte-standalone-app.js` explicitly delegates to it as the single battle owner.

## Current problem

The live signal screen is still generated by `databyte-standalone-app.js`, specifically its `signal()` flow. The visible button is currently capture-first:

```text
Launch DataByteCoin
Return / Scanner
```

Older battle modules and V3 bridge files are not the source of truth if the active signal screen never delegates to them.

## Canonical encounter flow target

The intended direct OS flow is now:

```text
Scanner
  ↓
Signal Found
  ↓
Battle Signal
  ↓
Battle Loop
  ├─ Attack
  ├─ Shield
  ├─ Change
  ├─ Item
  ├─ Return
  └─ Throw DataByteCoin
        ↓
      Capture Result
        ↓
      Scanner
```

`Launch DataByteCoin` may remain as a quick-capture option later, but it should not be the only visible or dominant action while battle is being rebuilt.

## Phase 2 implementation target

Modify `assets/js/databyte-standalone-app.js` directly:

1. Promote `Battle Signal` into the active signal action area.
2. Add a canonical `battle()` screen/state inside the standalone app or delegate to exactly one battle module.
3. Keep capture math inside the same encounter state.
4. Remove reliance on bridge overlays for the normal path.
5. After direct battle works, disable or delete deprecated patch stack entries from `dd-scan-bg.js`.

## Development rule

No future battle patch is complete unless the active signal button path in `databyte-standalone-app.js` enters the intended battle screen without relying on hidden overlays, duplicate launchers, or bridge fallbacks.
